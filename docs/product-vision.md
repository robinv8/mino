# Mino - 产品愿景文档

## 一句话定位

开源的通用 Agent 交互客户端——把 Agent 当成联系人，通过对话驱动一切。

## 核心洞察

1. 现有 IM 平台（Telegram、飞书等）支持的内容格式有限，不支持 Q&A、任务执行进度等结构化交互
2. 现有 IM 的消息是"死的"，而真实沟通中很多信息天然是结构化、可交互的
3. Agent 时代爆发，但交互层极度缺失——大家还在用聊天框打字
4. IM 不只是人与人沟通的工具，而是人与 Agent 协作的操作台

## 产品定位

Mino 是一个统一的界面，通过对话驱动一切。

类比：浏览器之于网站 = Mino 之于 Agent

## 核心设计原则

### Agent 就是联系人

- Agent 和人在对话列表里并列，不做特殊区分
- 用户发消息的方式完全一样
- 差异不在对话入口，而在消息内容的丰富度
- UI 统一，能力按消息类型自动渲染

### 消息是有生命周期的

- 消息不是"发出即定型"的静态记录
- 消息可以有状态变化（进度更新、状态流转）
- 消息可以被交互（确认、选择、操作）

## 对话对象

| 联系人 | 能做什么 |
|--------|---------|
| OpenClaw Agent | 自动化任务、信息查询 |
| Claude Code | 写代码、改 Bug、重构项目 |
| Cursor Agent | 项目开发 |
| CI/CD Bot | 看构建状态、触发部署 |
| 监控 Agent | 查看服务器状态 |
| 人（未来） | 普通聊天 |

## 消息类型

### 人发的消息
- 文本
- 图片/文件

### Agent 发的消息（在人的基础上扩展）
- 流式输出（打字机效果）
- 进度状态（执行中/完成/失败）
- 工具调用（可折叠展示）
- 确认请求（按钮交互）
- 结构化结果（表格/代码/图表）

## 架构思路

```
用户 ←→ Mino 客户端 ←→ Agent 平台（OpenClaw 等）←→ 工具/API/模型
```

协议适配层是核心，定义统一的交互原语：

| 交互原语 | 说明 |
|----------|------|
| text | 纯文本/流式输出 |
| progress | 任务进度，可更新状态 |
| confirmation | Agent 请求人类确认 |
| form | Agent 请求人类输入结构化数据 |
| result | 结构化结果展示 |
| tool_call | Agent 调用工具的可视化 |
| error | 错误/异常展示 |
| branch | 多路径选择 |

## 技术选型

- **原生客户端**：SwiftUI（macOS 先行，后续 iOS 共享代码）
- **先 macOS，再跨平台**：后续用 Kotlin/其他技术做 Android 和 Windows
- **首个对接平台**：OpenClaw（通过 ACP 协议 / WebSocket）
- **人与人通信**：未来对接 Matrix 协议（去中心化、开源、成熟）

## MVP 路线

### 第一步：OpenClaw 聊通
- 联系人列表（Agent）
- 一对一对话（人 ↔ Agent）
- 对接 OpenClaw（ACP 协议）
- 消息类型：文本、流式输出、进度、工具调用、确认请求、结构化结果
- 本地持久化（聊天记录 + Agent 配置）

### 第二步：接入 Claude Code
- Claude Code 作为另一个 Agent adapter
- 补充代码渲染、diff 展示等开发相关组件
- 项目上下文管理

### 第三步：人与人通信
- 对接 Matrix 协议
- 群聊（人 + Agent 混合）

### MVP 不做
- 人对人聊天
- 群聊
- 多平台（只做 macOS）
- 云同步

## 竞品参考

|------|--------|
| Beeper | 多平台聚合，桥接架构 |

| Element | 开源运营，Matrix 协议 |

## 已确认决策

- **对话模式**：UI 上是连续的消息流（微信式），底层按会话分段存储。用户无需感知"新建对话"，后端根据 Agent 平台的上下文限制自动分段
- **本地持久化**：聊天记录和 Agent 配置都存本地
- **Agent 添加方式**：MVP 手动填 URL
